# AGENTS.md — cmux-workshop 표준 협업 규범

이 파일은 `project-agent`와 `project-reload`가 프로젝트 루트로 복사하는
표준 hand-off 문서다. 모든 specialist CLI(Codex, Claude, Gemini 등)는
작업 시작 전에 이 파일과 자신에게 주입된 persona 파일을 함께 읽고, 충돌 시
이 파일의 공통 규범을 우선한다.

## 역할 분담

- Caller/Orchestrator는 목표, 제약, 현재 상태, 담당 agent, 완료 기준을 정리해
  각 specialist에게 전달한다.
- Specialist는 자신에게 배정된 범위만 수정하고, 다른 agent의 파일이나 변경을
  되돌리지 않는다.
- Reviewer/Verifier는 구현자가 남긴 결과와 검증 로그를 기준으로 위험, 누락,
  재현 절차를 보고한다. 직접 수정이 필요한 경우 수정 범위를 먼저 밝힌다.

## Hand-off 입력 형식

새 작업을 넘길 때는 아래 순서를 유지한다.

```markdown
## 목표
한 문장으로 완료 상태를 정의한다.

## 담당 범위
수정/조사할 파일, 모듈, 명령을 구체적으로 적는다.

## 제약
수정 금지 영역, 설치 금지, 네트워크 금지, naming/version 규칙 등을 적는다.

## 현재 상태
이미 완료된 작업, 실패한 명령, 관찰된 로그를 요약한다.

## 완료 보고
기대하는 출력 형식과 반드시 포함할 검증 결과를 적는다.
```

## 완료 보고 형식

각 agent는 작업 종료 시 다음 항목을 간결하게 보고한다.

- `변경`: 수정한 파일과 핵심 동작 변화
- `검증`: 실행한 명령과 통과/실패 결과
- `잔여 위험`: 해결하지 않은 문제, 가정, 다음 agent가 봐야 할 부분
- `인계`: 다음 담당자가 이어서 할 구체적 행동

## 안전 규칙

- destructive command(`rm -rf`, 강제 push, DB drop, 대량 삭제)는 caller가
  명시한 경우에만 실행하고, 대상 경로를 먼저 확인한다.
- dependency 설치(`npm install`, `brew install`, `pip install`)는 사용자가
  명시적으로 허용하지 않으면 실행하지 않는다.
- vendored snapshot, generated artifact, lockstep version 규칙이 주어진 경우
  그 제약을 우선한다.
- 작업 도중 예상 밖의 사용자 변경을 발견하면 되돌리지 말고, 관련 여부를 판단해
  함께 보존하는 방향으로 진행한다.

## 에스컬레이션 기준

아래 상황에서는 진행을 멈추고 caller에게 구체적으로 보고한다.

- 요구사항끼리 충돌해 동시에 만족할 수 없음
- 수정 금지 영역 없이는 목표 달성이 불가능함
- 검증 명령이 환경 문제인지 코드 문제인지 분리되지 않음
- 보안/데이터 손실 위험이 명확하고 사용자의 승인 범위를 벗어남
