# cmux-workshop 코드 리뷰 (Codex)
생성: 20260426-1210, 모델: OpenAI Codex

## TL;DR (3~5줄)
- 방향성은 좋다. cmux를 새로 감싸기보다 얇은 플러그인 레이어, SQLite 상태 저장, vendor된 모니터 런타임으로 묶은 선택은 프로젝트 정체성과 잘 맞는다.
- 다만 현재 `/project-view`는 핵심 경로에 두 개의 치명적 통합 결함이 있다: 의존성 체크가 문서대로 설치해도 실패할 수 있고, proxy `status`를 활성 판정으로 써서 실제 inject가 건너뛰어진다.
- DB 스키마와 스킬 문서가 일부 어긋난다. 특히 `local_surfaces.status='error'`를 문서/워크플로우는 쓰지만 스키마가 금지한다.
- "slash command"로 홍보한 `/project-*` 명령과 실제 `commands/` 구성의 차이를 정리해야 DX 신뢰도가 올라간다.
- 다음 버전은 `/project-view-stop`, port/PID 소유권 검증, schema migration을 우선 보강하는 것이 좋다.

## 강점 (top 5)
1. **일관된 워크플로우 모델** — `init → agent → reload → view`를 전면에 내세운 흐름이 README-ko의 문제 정의와 잘 연결된다 (`README-ko.md:7-9`, `README-ko.md:63-69`).
2. **Two-zone SQLite 설계** — portable 상태와 machine-local runtime ID를 테이블 prefix로 분리한 결정은 단일 파일 단순성과 재현성을 잘 절충한다 (`README-ko.md:544-551`, `plugins/cmux-workshop/tools/schema.sql:93-115`).
3. **vendor 전략이 명확함** — cmux-monitor 런타임을 `skills/project-view/runtime/`에 고정하고 re-vendor 절차를 문서화해 외부 경로 의존을 줄였다 (`CLAUDE.md:78-98`).
4. **Bash 기본 품질이 준수함** — 주요 신규 런처/DB 스크립트에 `set -euo pipefail`, 공통 helper, PID/log prefix가 적용되어 있다 (`plugins/cmux-workshop/skills/project-view/scripts/helpers.sh:6-22`, `plugins/cmux-workshop/tools/db.sh:19-25`).
5. **상태 진입점이 좋음** — `project-status`가 phase와 runtime status를 분리해 보여주고 다음 행동을 안내하는 구조는 장기 워크플로우에 유용하다 (`plugins/cmux-workshop/skills/project-status/SKILL.md:97-108`, `plugins/cmux-workshop/skills/project-status/SKILL.md:260-271`).

## 개선 영역

### [P1] `/project-view`가 proxy를 실제로 inject하지 못할 수 있음
- 위치: `plugins/cmux-workshop/skills/project-view/scripts/start.sh:18-25`, `plugins/cmux-workshop/skills/project-view/runtime/cmux-proxy.sh:414-494`
- 근거: `start.sh`는 `./cmux-proxy.sh status`의 exit code로 proxy 활성 여부를 판단한다. 그러나 `status` 명령은 "프록시 실행 중 아님"을 출력해도 정상 종료하는 표시용 명령이므로, cmux가 응답하는 환경에서는 `inject` 경로가 사실상 건너뛰어진다. 그 결과 dashboard는 켜져도 JSON-RPC 트래픽이 Redis Streams로 미러링되지 않아 핵심 관찰 기능이 빈 화면이 될 수 있다.
- 제안: `cmux-proxy.sh is-active` 같은 기계 판정용 subcommand를 추가해 PID 생존 + listen/upstream socket 상태 + `test_socket`을 검사하고, `start.sh`는 그 명령만 사용한다. 또는 `status --quiet`가 비활성 시 non-zero를 반환하도록 분리한다. 사람용 `status`의 출력 semantics와 스크립트용 readiness semantics를 섞지 않는 것이 핵심이다.

### [P1] 문서대로 설치해도 web dependency check가 실패할 수 있음
- 위치: `plugins/cmux-workshop/skills/project-view/scripts/check-deps.sh:44-49`, `plugins/cmux-workshop/skills/project-view/runtime/web/package.json:5-7`, `README.md:70-77`
- 근거: 문서는 `(cd runtime/web && npm run install:all)`을 안내하고, root `web/package.json`의 `install:all`은 `server`와 `client`에서만 `npm install`을 실행한다. 그런데 `check-deps.sh`는 `$WEB_DIR/node_modules`까지 필수로 검사한다. root web package에는 dependencies가 없으므로 문서대로 수행하면 root `node_modules`가 생성되지 않아 `/project-view`가 계속 실패할 수 있다.
- 제안: `WEB_DIR/node_modules` 검사를 제거하고 `server/node_modules`, `client/node_modules`만 확인한다. root에도 dependency가 필요하다면 `install:all` 앞에 `npm install`을 추가하고 root `package-lock.json`을 vendor에 포함한다. 현재 구조에서는 제거가 더 단순하다.

### [P1] `error` runtime status가 스키마 CHECK에 막힘
- 위치: `plugins/cmux-workshop/tools/schema.sql:104-110`, `plugins/cmux-workshop/skills/project-reload/SKILL.md:317-326`, `plugins/cmux-workshop/skills/project-status/SKILL.md:248-249`
- 근거: `project-reload` v0.5+ 흐름은 readiness 검증에서 인증 실패/command not found 등을 감지하면 `local_surfaces.status='error'`를 기록한다고 설명한다. 하지만 실제 DDL의 CHECK는 `'running'|'stopped'|'skipped'`만 허용한다. 에러 pane을 기록하는 순간 SQLite constraint failure가 발생하거나, 구현자가 문서와 달리 running으로 낮춰 기록하게 되어 `project-status`가 실패 원인을 보여줄 수 없다.
- 제안: `local_surfaces.status` CHECK와 `insert-surface.sql` 주석/SCHEMA.md를 `'running','stopped','skipped','error'`로 맞춘다. 이미 배포된 DB를 고려해 schema version 2 migration도 같이 정의한다.

### [P1] FK cascade가 실제 `db.sh exec/run`에서 보장되지 않음
- 위치: `plugins/cmux-workshop/tools/schema.sql:9`, `plugins/cmux-workshop/tools/schema.sql:59-64`, `plugins/cmux-workshop/tools/db.sh:56-94`, `plugins/cmux-workshop/tools/SCHEMA.md:121-127`
- 근거: SQLite의 `PRAGMA foreign_keys=ON`은 connection-local이다. `schema.sql`에서 한 번 켠 값은 이후 `db.sh exec`, `db.sh run`, `db.sh json`의 새 sqlite3 프로세스에 자동으로 유지되지 않는다. 그래서 설계 문서가 기대하는 `ON DELETE CASCADE`가 일반 삭제 경로에서는 동작하지 않을 수 있고, `agents` 삭제 후 `layout_splits`/`local_surfaces`가 orphan으로 남을 수 있다.
- 제안: `db.sh`의 모든 sqlite3 호출 앞에 `PRAGMA foreign_keys=ON;`을 주입한다. 예: `cmd_exec`는 `sqlite3 "$DB_PATH" "PRAGMA foreign_keys=ON; $sql"`, `cmd_run`은 temp stream에 pragma를 prepend한다. SCHEMA.md의 "persist per connection" 설명도 "매 connection마다 켜야 한다"로 수정한다.

### [P1] 광고한 `/project-*` slash command와 실제 commands 구성이 불일치함
- 위치: `README-ko.md:7-9`, `README.md:56-60`, `README.md:102-108`, `plugins/cmux-workshop/commands/`
- 근거: README는 `/project-init`, `/project-agent`, `/project-reload`, `/project-view`를 핵심 slash command로 설명하지만 실제 `commands/`에는 `code-quality`, `code-explore`, `merge-permissions`만 있다. Claude Code가 skill name을 자동 slash command로 노출하지 않는 환경이면 사용자는 문서의 첫 명령부터 실행할 수 없다.
- 제안: 각 핵심 entrypoint에 얇은 command shim을 추가한다. 예: `commands/project-view.md`는 `bash "${CLAUDE_PLUGIN_ROOT}/skills/project-view/scripts/start.sh"`를 실행하고 READY URL을 open하도록 지시한다. `project-init/agent/reload/status/reset`도 해당 skill 호출을 명시하는 shim을 두거나, README에서 "slash command" 표현을 "skill trigger"로 낮춘다.

### [P2] port 5173/3001 충돌과 PID 재사용을 런처가 충분히 구분하지 못함
- 위치: `plugins/cmux-workshop/skills/project-view/scripts/helpers.sh:12-18`, `plugins/cmux-workshop/skills/project-view/scripts/start.sh:27-51`, `plugins/cmux-workshop/skills/project-view/runtime/web/client/vite.config.js:6-22`, `plugins/cmux-workshop/skills/project-view/references/troubleshooting.md:38-44`
- 근거: `is_pid_alive`는 PID 생존만 확인하므로 `/tmp` PID가 재사용된 경우 다른 프로세스를 dashboard로 오인할 수 있다. 또한 Vite는 기본적으로 5173이 바쁘면 다음 포트로 올라갈 수 있는데 `start.sh`는 고정 URL `http://localhost:5173`만 health check한다. 5173에 다른 앱이 이미 있으면 wrong-ready가 될 수 있고, 3001 충돌은 browser open 후 disconnected로 드러날 수 있다.
- 제안: 시작 전 `lsof -nP -iTCP:5173 -sTCP:LISTEN` 및 3001 점유자를 확인하고, PID 파일은 command line/cwd가 `runtime/web`/`polling_monitor.py`인지 검증한다. Vite에는 `server.port=5173`, `server.strictPort=true`를 설정하거나 실제 출력 URL을 파싱해 sentinel에 반영한다.

### [P2] `/project-view-stop` 부재는 이제 YAGNI보다 운영 리스크가 큼
- 위치: `plugins/cmux-workshop/skills/project-view/SKILL.md:64-74`, `README.md:79-87`, `CLAUDE.md:140-146`
- 근거: `/project-view`는 단순 웹 서버뿐 아니라 cmux socket을 rename/inject하고 polling/background process를 남긴다. 종료가 kill snippet + proxy stop 수동 절차에 머물면 stale proxy/socket/PID가 남아 다음 호출을 망가뜨릴 가능성이 높다. 특히 관찰 도구는 데모/디버깅 중 반복적으로 켜고 끄게 된다.
- 제안: surface area를 크게 늘리지 않는 선에서 `project-view-stop` 하나만 추가한다. 구현은 `kill if owned`, `cmux-proxy.sh stop`, PID/log 정리, `cmux ping` 검증까지 idempotent하게 수행한다. 별도 status command가 부담이면 stop 결과에 상태 요약을 포함한다.

### [P2] AGENTS.md bootstrap 문서가 있지만 실제 표준 사본이 없음
- 위치: `plugins/cmux-workshop/skills/project-agent/SKILL.md:251-277`, `plugins/cmux-workshop/skills/project-reload/SKILL.md:67-85`, 영역: repo root / `plugins/cmux-workshop/AGENTS.md`
- 근거: `project-agent`와 `project-reload`는 플러그인의 표준 `AGENTS.md`를 프로젝트 루트로 복사해 모든 specialist CLI가 공통 규범을 읽도록 설계했다. 그러나 현재 저장소에는 root `AGENTS.md`나 `plugins/cmux-workshop/AGENTS.md`가 없다. 결과적으로 매번 경고 후 skip되고, 문서가 강조한 hand-off/report 규범이 실제로는 배포되지 않는다.
- 제안: 최소 표준 `AGENTS.md`를 플러그인에 포함하고 re-vendor/rename 절차에도 보존 대상을 명시한다. 없다면 SKILL.md에서 bootstrap을 제거하고 각 persona에 공통 규범을 직접 포함한다.

### [P2] hook이 정상 워크플로우를 막거나 조용히 무력화될 수 있음
- 위치: `plugins/cmux-workshop/hooks/hooks.json:3-14`, `plugins/cmux-workshop/hooks/scripts/block-dangerous.sh:1-23`, `plugins/cmux-workshop/hooks/scripts/save-conv-before-commit.sh:60-65`
- 근거: 모든 Bash에 hook을 거는 전략은 안전망으로 좋지만, `block-dangerous.sh`는 `jq` 의존성이 없을 때 조용히 COMMAND가 비어 우회될 수 있고, regex 기반 `rm` 파싱은 quote/`--`/변수 확장/여러 명령 조합에서 오탐·미탐 여지가 크다. 반대로 commit hook은 180초를 체크하면서 메시지는 "older than 1 minute"라고 해 UX 혼란을 만든다.
- 제안: hook 공통 preflight에서 `jq`를 필수 확인하고 실패 시 deny 또는 명확한 경고를 반환한다. 위험 명령 파서는 최소한 shell 토큰화를 Python `shlex`로 수행하고, allow/deny 정책을 문서와 테스트 케이스로 고정한다. commit stale 메시지는 실제 TTL과 일치시킨다.

### [P2] manifest 설명에 legacy 이름과 제거된 기능이 남아 있음
- 위치: `plugins/cmux-workshop/.claude-plugin/plugin.json:4`, `.claude-plugin/marketplace.json:4-12`, `CLAUDE.md:46-63`, `CLAUDE.md:126-127`
- 근거: naming convention은 `/project-view`가 headline이고 `/cmux-workshop`이 아니라고 설명하지만 plugin description은 `/cmux-workshop one-shot monitor launcher`라고 쓴다. 또한 `tdd-team`은 의도적으로 제외했다고 CLAUDE.md가 말하지만 marketplace/plugin 설명과 keyword에는 TDD가 남아 있다. 설치 전 사용자가 보는 가장 중요한 metadata가 실제 기능과 다르다.
- 제안: manifest description/keywords에서 `/cmux-workshop`, `TDD team`, `tdd`를 제거하고 README의 skill catalog와 맞춘다. 버전 lockstep 규칙에 따라 marketplace와 plugin.json을 함께 갱신한다.

### [P2] SQLite 단일 파일에 portable/local을 섞는 운영 규칙이 자동화되어 있지 않음
- 위치: `README-ko.md:544-551`, `plugins/cmux-workshop/tools/README.md:65-76`, `plugins/cmux-workshop/tools/schema.sql:68-85`, `plugins/cmux-workshop/tools/schema.sql:93-115`
- 근거: Two-zone을 한 DB에 두는 선택은 단순하지만, `project_info.project_root` 같은 절대 경로와 `local_*` runtime ID가 같은 파일에 들어간다. 문서는 commit 전 `reset-local.sql`을 권장하지만 강제되지 않아 다른 머신에서 stale workspace/surface 상태가 공유될 수 있다.
- 제안: `tools/db.sh scrub-local` 또는 `project-status`의 "commit-ready" 체크를 추가해 local zone과 absolute path를 정리/재캡처하도록 안내한다. 장기적으로는 portable export/import(`project.db` → JSON/SQL dump)나 `.claude/project.local.db` 분리를 검토한다.

### [P2] schema version은 있지만 migration 전략이 아직 없음
- 위치: `plugins/cmux-workshop/tools/schema.sql:17-24`, `plugins/cmux-workshop/tools/SCHEMA.md:11-18`, `CLAUDE.md:149-168`
- 근거: `project.schema_version`은 예약되어 있지만 `db.sh init`은 `CREATE TABLE IF NOT EXISTS`만 수행한다. 이번 리뷰의 `status='error'` 같은 제약 변경은 기존 DB에서 단순 init으로 반영되지 않는다. 플러그인을 실제 프로젝트들에 적용하기 시작하면 schema drift가 바로 운영 문제가 된다.
- 제안: `tools/migrations/001.sql`, `002.sql` 형태와 `metadata.schema_version` 또는 `project.schema_version` 기반의 `db.sh migrate`를 추가한다. 각 project-* 스킬의 bootstrap 단계는 `init` 대신 `migrate`를 호출하도록 통일한다.

## 위험 분석
- **stale PID / PID reuse**: 현재 helper는 `kill -0`만 본다. PID 파일이 다른 프로세스를 가리키면 web/polling을 재사용한다고 오판한다. command line과 cwd 검증이 필요하다.
- **port 5173/3001 충돌**: Vite/Express 둘 중 하나라도 충돌하면 "READY"가 잘못된 서비스에 붙거나 브라우저에서 disconnected가 난다. strictPort와 사전 포트 점유자 확인이 필요하다.
- **cmux socket 부재/변형**: check-deps는 socket 존재를 보지만 start는 proxy 상태 판정이 부정확하다. inject 실패 시 원복은 vendor script에 있으나, launcher가 active 상태를 검증해야 한다.
- **Redis 다운**: check-deps가 Redis를 잘 감지하지만, Redis가 start 이후 죽으면 web/polling/proxy가 부분 실패한다. dashboard에 Redis disconnected 상태와 재시도 안내가 있으면 좋다.
- **web/node_modules 미설치**: 현재 root node_modules 검사 때문에 문서대로 설치한 사용자도 실패할 수 있다. 즉시 수정해야 한다.
- **hooks 정상 워크플로우 영향**: 모든 Bash에 걸리는 hook은 안전하지만, jq 누락/regex 오탐/commit stale 정책 때문에 작업 흐름을 막거나 반대로 우회될 수 있다. hook은 테스트가 필요하다.
- **FK cascade 부작용/미작동**: 의도한 cascade는 각 sqlite connection에서 foreign_keys를 켜지 않으면 작동하지 않는다. 반대로 migration 없이 CHECK를 바꾸면 기존 DB는 계속 낡은 제약을 가진다.
- **vendor drift**: runtime은 외부 스냅샷이라 직접 수정 금지 원칙이 좋지만, start/check-deps와 runtime behavior 사이의 계약 테스트가 없으면 re-vendor 때 쉽게 깨진다.

## 추천 다음 단계 (우선순위 순서)
1. `project-view` launch path를 고친다: proxy active 판정 subcommand 추가, web dependency check 수정, port/PID 검증 추가.
2. DB wrapper를 수정해 모든 sqlite3 connection에서 `PRAGMA foreign_keys=ON`을 강제하고, `local_surfaces.status`에 `error`를 추가한다.
3. `/project-*`가 실제 slash command인지 결정한다. 맞다면 command shim 6개를 추가하고, 아니라면 README/manifest의 표현을 skill trigger로 통일한다.
4. `/project-view-stop`을 추가해 web/polling/proxy/socket 원복을 idempotent하게 제공한다.
5. 표준 `AGENTS.md`를 플러그인에 포함하거나 bootstrap 설계를 제거한다.
6. hooks에 `jq` preflight와 shlex 기반 최소 parser/test fixtures를 붙인다.
7. `tools/migrations/`와 `db.sh migrate`를 도입해 schema 변경을 안전하게 배포한다.
8. CI 또는 최소 검증 스크립트를 추가한다: JSON validate, bash syntax, check-deps mock, db FK cascade test, `/project-view` dry-run contract test.
